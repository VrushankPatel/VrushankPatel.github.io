Latency Engineering

Low Latency Order Routing

Sub-100µs order routing for equity trading: from kernel bypass to JVM heap elimination on the critical path.

DPDKZGCLock-Free QueuesCPU Affinity

Problem

A market-making desk required order-to-wire latency under 100µs at p99. The existing stack used standard Linux networking and JVM heap patterns causing multi-millisecond GC pauses under load.

Key Metrics

p50 latency (before)340µs
p50 latency (after)48µs
p99 latency (after)91µs
GC pause reduction99.7%

Approach

  • Migrated from standard sockets to DPDK-based kernel bypass for NIC-to-application path
  • Adopted object pooling and off-heap memory to eliminate GC pressure on hot paths
  • Replaced ConcurrentHashMap with a lock-free open-addressing table
  • Pinned threads to isolated cores with NUMA-aware allocation
  • Tuned ZGC with compact pause targets for a 16 GB capital markets heap

Tradeoffs

  • DPDK requires dedicated NIC hardware and significantly raises operator expertise bar
  • Off-heap memory management reintroduces manual lifecycle complexity
  • Thread pinning reduces throughput headroom for burst capacity
  • Lock-free structures are harder to reason about during incident debugging

JVM Internals

JVM Memory & GC Optimization

Eliminating stop-the-world pauses in a latency-sensitive trading application via heap archaeology and GC log analysis.

ZGCG1GCasync-profilerHeap Analysis

Problem

A risk calculation service was experiencing unpredictable 300ms GC pauses during peak market hours. The pauses caused downstream timeout cascades and missed SLA windows.

Key Metrics

Max GC pause (before)340ms
Max GC pause (after)2.1ms
Allocation rate reduction68%
Heap size24 GB

Approach

  • Used async-profiler in allocation mode to identify hot allocation sites
  • Replaced short-lived churn with object pool patterns on critical paths
  • Migrated from G1GC to ZGC; tuned concurrent GC threads and region sizes
  • Fixed humongous object allocations causing premature old-gen promotion
  • Reduced live set by 40% by fixing reference retention bugs via heap dump analysis

Tradeoffs

  • ZGC trades higher memory overhead for lower latency guarantees
  • Object pooling increases code complexity and introduces reset-bug risk
  • Aggressive GC tuning is JVM-version sensitive; revisit on every major upgrade

Kubernetes

Stateful Workloads on Kubernetes

Running Kafka and stateful trading services on Kubernetes: storage class selection, graceful pod disruption, and broker identity preservation.

StatefulSetsPodDisruptionBudgetCSIKafka on K8s

Problem

Running a Kafka cluster on Kubernetes while maintaining ordering guarantees, handling node drains gracefully, and preserving broker-to-partition affinity during rolling upgrades.

Key Metrics

Rolling upgrade downtime0 ms
Storage IOPS (local NVMe)450K
Post-restart rebalance< 8 s

Approach

  • StatefulSets with stable pod DNS for deterministic broker identity
  • PodDisruptionBudgets enforcing max-unavailable during node drains
  • Local NVMe via CSI driver over network storage for I/O latency SLOs
  • pre-stop lifecycle hooks for graceful leader handoff before SIGTERM
  • topologySpreadConstraints to avoid broker co-location on same failure domain

Tradeoffs

  • Local NVMe ties pods to specific nodes, complicating rescheduling
  • StatefulSet rollouts are serial and significantly slower than Deployment rollouts
  • PDB constraints can block cluster-autoscaler from draining nodes

Messaging Architecture

Message Queue Architecture at Scale

Durable, ordered, multi-consumer event streaming for high-frequency market data distribution at 2M+ events per second.

KafkaConsumer GroupsCompacted TopicsExactly-Once

Problem

Fan out 2M+ events/second to 40+ consumer groups with per-instrument ordering guarantees, at-least-once semantics with idempotent consumers, and sub-5ms end-to-end latency.

Key Metrics

Throughput2.4M msg/s
p99 e2e latency3.8ms
Consumer groups43
Partition count480

Approach

  • Partitioned by instrument ID for strict ordering within each instrument stream
  • Compacted topics for latest-state snapshots alongside raw event topics
  • 9-broker cluster with rack-aware replication (RF=3, min.insync=2)
  • Tuned linger.ms and batch.size for the throughput/latency tradeoff
  • Consumer-side idempotency keys to tolerate duplicate delivery

Tradeoffs

  • Instrument-keyed partitioning creates hot partitions for high-volume tickers
  • Large partition counts increase coordinator overhead and rebalance duration
  • Kafka transactions (exactly-once) add ~15% latency overhead on producers

Distributed Systems

Distributed System Tradeoffs

A practical framework for navigating consistency, availability, and partition tolerance in production systems — beyond the CAP theorem abstraction.

CAP / PACELCRaftLinearizabilityHLC

Framework

CAP is a useful mental model but insufficient for production decisions. PACELC provides better guidance: during partitions choose between Availability and Consistency; otherwise choose between Latency and Consistency. Most real systems decide based on data criticality, not theology.

Decision Matrix

Order stateLinearizable
Market dataEventual OK
Risk limitsStrong + Fence
Audit logsDurable ordered

Key Insights

  • Raft provides linearizable reads only when routing through the leader; followers are stale by design
  • Fencing tokens on distributed locks prevent split-brain writes during network partitions
  • Monotonic read guarantees are sufficient for most UI and reporting workloads
  • Clock skew limits distributed timestamp ordering; prefer logical clocks or HLCs

Common Failure Modes

  • Assuming eventual consistency is safe for financial position calculations
  • Using wall-clock time for ordering in distributed systems without NTP discipline
  • Ignoring split-brain scenarios in active-active database setups
  • Conflating idempotency with exactly-once delivery semantics

Reliability

Failure-Oriented Design

Building systems where failure is a first-class design input. From chaos engineering to SLO-driven architecture decisions.

SLOsChaos EngineeringCircuit BreakersBulkheads

Philosophy

Reliability is not a feature added at the end of a project. It is an emergent property of accumulated design decisions. Every component should have a documented failure mode, a defined blast radius, and a recovery mechanism that has been tested under realistic conditions.

SLO Architecture

Availability target99.95%
Error budget / month21.9 min
Latency SLO (p95)< 200ms
MTTR target< 5 min

Techniques

  • Circuit breakers with half-open probing to prevent thundering herd on recovery
  • Bulkhead isolation per consumer to contain failure blast radius
  • Timeout hierarchy: client timeout > server timeout > dependency timeout
  • Backpressure signaling from downstream to upstream via reactive streams
  • Chaos injection in staging: pod kills, network latency, disk failures

Key Lessons

  • Retry storms are often worse than the original failure they respond to
  • The most dangerous failures are silent ones: data corruption, partial writes
  • SLOs drive architectural decisions better than vague uptime aspirations
  • Runbooks rot; prefer automation and self-healing over documentation

Current Focus

What I am Exploring

eBPF — Kernel observability and network policy enforcement without kernel modules or service mesh overhead

Linux Scheduler Internals — CFS, task groups, and the impact of NUMA topology on tail latency

JVM Vector API — SIMD intrinsics through Java's Panama project for numeric compute workloads

Kubernetes Gateway API — The next generation of traffic management beyond Ingress

Deterministic Simulation Testing — FoundationDB-style correctness verification for distributed systems

My current learning is oriented toward the intersection of kernel-level observability and application-level performance. eBPF represents a fundamental shift in how we can instrument production systems without performance overhead or invasive code changes.

I am also deeply interested in deterministic simulation testing, a technique pioneered by FoundationDB where an entire distributed system is simulated deterministically, allowing bugs that would take months to surface in production to be reproduced in seconds.

These two directions represent the same underlying thesis: the future of reliability engineering is a shift from reactive observability toward proactive correctness verification.
